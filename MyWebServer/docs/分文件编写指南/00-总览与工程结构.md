## 一、为什么学完了还要分文件

### 1.1 单文件 step11.cpp 的问题

| 问题 | 说明 |
|------|------|
| **太长** | 600+ 行全在一个文件，改一处要找半天 |
| **职责混在一起** | 日志、MySQL、HTTP 解析、epoll 全挤在一起 |
| **不好协作** | 两人很难同时改同一个 cpp |
| **不像真实项目** | 交付 / 简历项目通常是多文件 + Makefile |

### 1.2 分文件不是「重写」，是「搬家」

```text
step11.cpp 里的代码块  →  剪贴到对应 .h / .cpp
逻辑不变，只是物理位置变了
```

你已经有能跑通的单文件版本，分文件阶段**不要顺便大改架构**，先保证编译通过、行为一致。

## 二、单文件归档 vs 多文件交付

```text
MyWebServer/
├── Step1to6/              ← 学习历程（保留，不删）
│   ├── Step1.cpp … Step5.cpp
│   └── step6.cpp …
├── step9.cpp step10.cpp step11.cpp   ← 后期单文件（保留）
│
├── src/                  ← 【交付工程】源码目录（.h / .cpp）
│   ├── main.cpp
│   ├── Log.h / Log.cpp
│   └── …
├── server                  ← 【编译产物】可执行文件（非目录！）
├── Makefile
├── www/
└── docs/分文件编写指南/     ← 本系列
```

| 路径 | 用途 |
|------|------|
| `Step1to6/`、`step11.cpp` | 复习、对照、答辩展示「学习路径」 |
| `src/` | 多文件源码，日常开发、提交 Git、写 README |
| `server`（根目录文件） | `make` 生成的进程，`./server` 启动 |

### 2.1 为什么源码放 `src/` 而不是 `server/`？（方法 B）

单文件学习阶段常用 `g++ -o server` 在**项目根目录**生成可执行文件 `server`。  
若再 `mkdir server` 建源码文件夹，**文件与目录不能同名**，`mkdir` 会失败。

本系列采用 **方法 B**：

| 名称 | 类型 | 含义 |
|------|------|------|
| `src/` | **目录** | 所有模块 `.h` / `.cpp` |
| `server` | **文件** | 编译产物，Makefile 中 `-o server` |
| `-Isrc` | 编译选项 | 头文件搜索路径，代码仍写 `#include "HttpParser.h"` |

> 若根目录已有旧的 `server` 可执行文件，先 `rm server` 或 `mv server server.bak`，再 `mkdir -p src`。

## 五、模块依赖关系图

```text
                    main.cpp
                       │
         ┌─────────────┼─────────────┐
         ▼             ▼             ▼
   ServerConfig    ThreadPool      Log
         │             │             │
         ▼             └──────┬──────┘
   HttpResponse              │
         │                    ▼
         ▼              HttpConnection ◄── EpollHelper
   HttpHandler                │
         │                    │
    FormParser                │
         │                    │
      SqlPool ─────────────────┘
         │
   HttpParser ──► HttpTypes.h
```

**编译顺序建议（从叶子到根）：**

1. `HttpTypes.h`  
2. `ServerConfig`  
3. `Log`、`ThreadPool`、`FormParser`  
4. `SqlPool`（依赖 Log）  
5. `EpollHelper`  
6. `HttpResponse`（依赖 ServerConfig）  
7. `HttpParser`（依赖 HttpTypes、HttpConnection 前向声明）  
8. `HttpHandler`（依赖 HttpResponse、FormParser、SqlPool）  
9. `HttpConnection`（依赖 Parser、Handler、ThreadPool、EpollHelper、Log）  
10. `main.cpp`

## 七、通用约定（必读）

### 7.1 头文件防护

每个 `.h` 第一行：

```cpp
#pragma once
```

### 7.2 编译位置与 include

在 **项目根目录**（与 `www/` 同级）：

```bash
make
./server
```

Makefile 使用 `-Isrc`，代码里：

```cpp
#include "HttpParser.h"
#include "Log.h"
```

### 7.3 `.h` 放什么、`.cpp` 放什么

| `.h` | `.cpp` |
|------|--------|
| 类声明、函数声明、`enum` | 函数实现、类成员函数实现 |
| `extern` 全局变量 | 全局变量定义 |

### 7.4 全局变量

| 变量 | 定义位置 | 声明位置 |
|------|----------|----------|
| `g_cfg` | `ServerConfig.cpp` | `ServerConfig.h` |
| `g_ready_queue` 等 | `HttpConnection.cpp` | `HttpConnection.h` |

### 7.5 常见编译错误

| 报错 | 原因 |
|------|------|
| `undefined reference to ...` | 漏加 `.cpp` 到 Makefile |
| `multiple definition` | 函数写在 `.h` 且被多个 cpp include，应放 `.cpp` 或 `inline` |
| `HttpRequest was not declared` | 漏 `#include "HttpTypes.h"` |

## 八、两种迁移路线

### 路线 A：正向跟写（推荐初学者）

从 [01-Step1](./01-Step1-main最小服务器.md) 开始，每步新建/修改文件，`make` 通过后再下一步。  
**优点：** 理解每步为什么拆。  
**缺点：** 耗时较长。

### 路线 B：从 step11.cpp 反向拆分（适合已熟代码）

1. 新建空 `src/` 目录。  
2. 打开 [11-Step11](./11-Step11-ServerConfig与Makefile.md) 文末 **「剪切对照表」**。  
3. 按模块剪贴，每剪一块 `make` 一次。  
**优点：** 快。  
**缺点：** 容易不知道模块边界。
