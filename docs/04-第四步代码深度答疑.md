## 问题1：`handleClient` 和 `handleRequest` 有什么区别？

### 相关代码

**第三篇（Step3）—— `handleRequest`：**

```cpp
string handleRequest(const HttpRequest& req) {
    if (req.method != "GET") return buildHttpResponse(405, ...);
    if (!isPathSafe(req.path)) return buildHttpResponse(403, ...);
    // pathToFile → readFile → getContentType → buildHttpResponse(200, ...)
}

// main 里：
accept → read → parseRequestLine → handleRequest(req) → write → close
```

**第四篇（Step4）—— 拆成两层：**

```cpp
string buildResponseFromRequest(const HttpRequest& req) {
    // 函数体和 handleRequest 相同
}

void handleClient(int epfd, int client_fd) {
    read(client_fd, buffer, ...);
    parseRequestLine(rawRequest, req);
    response = buildResponseFromRequest(req);  // 或 handleRequest(req)
    write(client_fd, response.c_str(), response.size());
    removeFd(epfd, client_fd);
    close(client_fd);
}
```

### 解答

#### 一句话

| 函数 | 管什么 |
|------|--------|
| **`handleRequest` / `buildResponseFromRequest`** | **厨房做菜**：根据 `HttpRequest` 生成 HTTP 响应**字符串** |
| **`handleClient`** | **服务员全流程**：读 socket → 解析 → 做菜 → 写 socket → 从 epoll 移除 → 关连接 |

**`handleClient` 内部会调用 `handleRequest`（或改名后的函数），但不是替代关系，是「外层包内层」。**

---

#### 输入输出对比

| | `handleRequest` | `handleClient` |
|---|----------------|----------------|
| **参数** | `HttpRequest`（已解析好的请求） | `epfd` + `client_fd`（epoll 实例 + socket 编号） |
| **返回值** | `string`（完整 HTTP 响应） | `void`（直接在函数里 write） |
| **是否碰网络** | ❌ 不 read/write | ✅ read + write |
| **是否碰 epoll** | ❌ | ✅ `removeFd` |
| **是否碰静态文件** | ✅ 核心逻辑在这 | ✅ 通过调用内层函数 |

---

#### 流程图

```text
【第三篇】一条线写在 main 里：

  accept
    → read
    → parseRequestLine → req
    → handleRequest(req)     ← 只生成 response 字符串
    → write
    → close


【第四篇】main 只管 epoll，I/O 进 handleClient：

  epoll_wait
    → 若是 listen_fd：accept，addFd(新 client_fd)
    → 若是 client_fd：handleClient(epfd, client_fd)
                          ├─ read
                          ├─ parseRequestLine
                          ├─ buildResponseFromRequest  ← 等同 handleRequest
                          ├─ write
                          ├─ removeFd
                          └─ close
```

---

#### 为什么第四篇要单独写 `handleClient`？

第三篇的 `main` 里，**read / parse / write / close** 和 **handleRequest** 写在一起。  
第四篇引入 epoll 后，每个 client_fd 就绪时要：

1. 读数据  
2. 业务处理（原来的 `handleRequest`）  
3. 写回  
4. **`epoll_ctl(DEL)`** —— 第三篇没有  
5. **`close`**  

把 1～5 收成 **`handleClient`**，`main` 里 epoll 循环就只写：

```cpp
if (fd == listen_fd) { accept... }
else { handleClient(epfd, fd); }
```

结构更清晰，也避免和两个 `handle*` 名字混淆（见 [04 指南 2.4 节](./04-epoll高并发指南.md#24-为什么把-handlerequest-改成-buildresponsefromrequest)）。

---

#### 小结

| 你的疑问 | 答案 |
|---------|------|
| 是不是同一个东西？ | **不是**。内层业务相同，外层 `handleClient` 多管 I/O 和 epoll |
| 能只写 `handleRequest` 吗？ | 可以，但 read/write/epoll 总要写在某处，第四篇放在 `handleClient` |
| 改名必须吗？ | 内层继续叫 `handleRequest` 完全可以 |

## 问题2：`setNonBlocking` 里的 `fcntl` 是什么？参数什么意思？

### 相关代码

```cpp
void setNonBlocking(int fd) {
    int flags = fcntl(fd, F_GETFL, 0);
    fcntl(fd, F_SETFL, flags | O_NONBLOCK);
}
```

### 解答

#### `fcntl` 是什么缩写？

| 拆解 | 含义 |
|------|------|
| **f** | **f**ile |
| **cntl** | **cntl**（control，控制） |

**`fcntl` = file control（文件控制）** —— Unix/Linux 里用来**查询或修改**已打开「文件描述符」属性的系统调用。  
socket 在 Linux 里也是 fd，所以 socket 也能用 `fcntl`。

---

#### 两句话在干什么？

```text
第 1 句：先读出 fd 当前有哪些「标志位」
第 2 句：在原有标志上加上「非阻塞」，再写回去
```

不能一上来就 `F_SETFL O_NONBLOCK`，否则会**覆盖掉**原来其它标志；所以要 **GET → 或上 O_NONBLOCK → SET**。

---

#### 第一次调用：`fcntl(fd, F_GETFL, 0)`

| 参数 | 含义 |
|------|------|
| `fd` | 要操作的 socket（listen_fd 或 client_fd） |
| `F_GETFL` | **Get FLags**：读取 fd 的状态标志 |
| `0` | 这里不用，填 0 |

**返回值 `flags`：** 当前标志的整数（位掩码），例如是否非阻塞、是否追加写等。

---

#### 第二次调用：`fcntl(fd, F_SETFL, flags | O_NONBLOCK)`

| 参数 | 含义 |
|------|------|
| `fd` | 同一个 socket |
| `F_SETFL` | **Set FLags**：设置 fd 的状态标志 |
| `flags \| O_NONBLOCK` | 在原有标志基础上，**加上**非阻塞 |

**`O_NONBLOCK` 是什么？**

| 名字 | 含义 |
|------|------|
| **O** | Option / Open 类前缀（开关选项） |
| **NONBLOCK** | 非阻塞 |

加上后：

| 操作 | 阻塞模式（默认） | 非阻塞模式 |
|------|----------------|-----------|
| `read` 没数据 | 进程**卡住等** | 立刻返回 `-1`，`errno == EAGAIN` |
| `accept` 没新连接 | 进程**卡住等** | 立刻返回 `-1`，`errno == EAGAIN` |

---

#### 为什么 Step4 需要非阻塞？

配合 **epoll** 的标准写法：

```text
epoll_wait 告诉你「这个 fd 可读」
    → 再去 read / accept
    → 若一次没读完，非阻塞 read 返回 EAGAIN，不会卡死整个服务器
    → listen_fd 上要 while(accept) 直到 EAGAIN，一次接完排队连接
```

**生活类比：**  
阻塞 = 站在门口一直等客人，别的桌全顾不上。  
非阻塞 = 门口看一眼，没有客人立刻回去大厅（epoll_wait）看哪桌举手。

---

#### 小结

| 问题 | 答案 |
|------|------|
| `fcntl` 是什么？ | 文件/ fd 控制函数 |
| 为何调用两次？ | 先 GET 再 SET，保留其它标志 |
| `O_NONBLOCK` 作用？ | read/accept 无数据时不阻塞整个进程 |
| 谁要设？ | **listen_fd** 和每个 **client_fd** 都要（accept 前/后加入 epoll 时） |

## 问题3：`addFd` 里的 `epoll_event`、`EPOLLIN`、`epoll_ctl` 是什么？

### 相关代码

```cpp
void addFd(int epfd, int fd) {
    epoll_event ev{};
    ev.events = EPOLLIN;
    ev.data.fd = fd;
    epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);
}
```

### 解答

#### 先理解 epoll 三件套

| API | 一句话 |
|-----|--------|
| `epoll_create1` | 创建一个 epoll **实例**（监视器），返回 `epfd` |
| `epoll_ctl` | 往监视名单 **增加 / 修改 / 删除** 某个 fd |
| `epoll_wait` | **等待**名单里有哪些 fd 就绪 |

`addFd` 封装的是 **`epoll_ctl` 的 ADD 操作**。

---

#### `epoll_event` 是什么？

**`epoll_event`** = epoll **事件**结构体，描述：

1. 你想监视这个 fd 的**哪种事件**（`events` 字段）  
2. 就绪时你想**带回什么信息**（`data` 字段，常用 `data.fd`）

```cpp
epoll_event ev{};   // {} 零初始化
ev.events = EPOLLIN;
ev.data.fd = fd;
```

| 字段 | 你的赋值 | 含义 |
|------|---------|------|
| `events` | `EPOLLIN` | 关心「可读」事件 |
| `data.fd` | `fd` | 等会儿 `epoll_wait` 返回时，告诉我**是哪个 fd** |

**`data` 还可以是别的类型**（如指针），本篇用最简单的 **`data.fd` 存整数 fd**。

---

#### `EPOLLIN` 是什么？

| 拆解 | 含义 |
|------|------|
| **EPOLL** | epoll 机制前缀 |
| **IN** | input，**输入/可读** |

| fd 类型 | `EPOLLIN` 就绪时表示 |
|---------|-------------------|
| **listen_fd** | 有新连接，可以 `accept` |
| **client_fd** | 有数据到达，可以 `read` |

常见还有（本篇未用）：

| 常量 | 含义 |
|------|------|
| `EPOLLOUT` | 可写 |
| `EPOLLERR` | 错误 |
| `EPOLLET` | 边缘触发（本篇用默认 LT，不加） |

---

#### `epoll_ctl` 四个参数详解

```cpp
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);
```

| 参数 | 你的代码 | 含义 |
|------|---------|------|
| 1 | `epfd` | epoll 实例（`epoll_create1` 的返回值） |
| 2 | `EPOLL_CTL_ADD` | 操作类型：**添加** fd 到监视名单 |
| 3 | `fd` | 要被监视的 socket（listen 或 client） |
| 4 | `&ev` | 指向 `epoll_event`，说明监视什么事件、带回什么 data |

**`EPOLL_CTL` 其它操作（对比记忆）：**

| 操作 | 何时用 |
|------|--------|
| `EPOLL_CTL_ADD` | 新 fd 加入 epoll（`addFd`） |
| `EPOLL_CTL_MOD` | 修改已存在 fd 的事件 |
| `EPOLL_CTL_DEL` | 从 epoll 移除（`removeFd`） |

---

#### 什么时候调用 `addFd`？

```text
main 启动后：addFd(epfd, listen_fd);     // 监视「新连接」

accept 成功后：addFd(epfd, client_fd);   // 监视「这个客人的数据」
```

---

#### 生活类比

| 概念 | 类比 |
|------|------|
| `epfd` | 大厅值班表 |
| `epoll_ctl ADD` | 把某一桌登记到值班表 |
| `EPOLLIN` | 「这桌举手了（有事）」 |
| `ev.data.fd` | 桌号（fd 编号） |
| `epoll_wait` | 值班员看哪几桌举手了 |

## 问题4：`removeFd` 在做什么？参数什么意思？

### 相关代码

```cpp
void removeFd(int epfd, int fd) {
    epoll_ctl(epfd, EPOLL_CTL_DEL, fd, nullptr);
}
```

在 `handleClient` 末尾调用：

```cpp
write(client_fd, response.c_str(), response.size());
removeFd(epfd, client_fd);
close(client_fd);
```

### 解答

#### 一句话

**连接处理完了，从 epoll 监视名单里摘掉这个 fd，再 `close`，避免 epoll 继续监视已关闭或无效的 fd。**

---

#### `epoll_ctl` 参数（DEL 版）

```cpp
epoll_ctl(epfd, EPOLL_CTL_DEL, fd, nullptr);
```

| 参数 | 含义 |
|------|------|
| `epfd` | epoll 实例 |
| `EPOLL_CTL_DEL` | **Delete**：从监视名单**删除**这个 fd |
| `fd` | 要删除的 client_fd |
| `nullptr` | DEL 时不需要 `epoll_event`，填空指针 |

---

#### 为什么必须先 `removeFd` 再 `close`？

| 顺序 | 说明 |
|------|------|
| 推荐 | `removeFd` → `close` |
| 若只 `close` 不 DEL | 有的系统上 epoll 里仍留着旧 fd 条目，可能**误唤醒**、**忙循环**，甚至行为未定义 |
| 本篇模型 | 短连接：一次请求 → 响应 → 关连接，所以**每次处理完 client 必 DEL** |

```text
handleClient 结束时的 fd 生命周期：

  曾在 epoll 里 EPOLLIN 监视
       ↓
  read / write 完成
       ↓
  epoll_ctl(DEL)   ← removeFd：从值班表划掉
       ↓
  close(fd)        ← 内核回收这个连接
```

**生活类比：** 客人离店后，先从值班表上**划掉桌号**，再**收拾桌子**（close）。否则服务员还可能跑去空桌。

---

#### `addFd` 和 `removeFd` 成对出现

| 时机 | 操作 |
|------|------|
| `accept` 得到 `client_fd` | `addFd(epfd, client_fd)` |
| `handleClient` 处理完毕 | `removeFd(epfd, client_fd)` + `close` |

**listen_fd** 一般只在程序退出前才关，整个运行期间一直留在 epoll 里，**不需要**每 accept 一次就 DEL listen_fd。

---

#### 小结

| 问题 | 答案 |
|------|------|
| `removeFd` 做什么？ | `epoll_ctl(DEL)`，取消监视 |
| 为何 `nullptr`？ | DEL 不需要传 `epoll_event` |
| 和 `close` 关系？ | 先 DEL 再 close，好习惯 |

## 问题5：相比第三步，`handleClient` 和 `main` 改了什么？为什么？

### 相关代码对照

**第三步 `main` 核心：**

```cpp
while (true) {
    int client_fd = accept(server_fd, ...);   // 阻塞等一个客人
    read(client_fd, buffer, ...);
    parseRequestLine → handleRequest(req);
    write(client_fd, response...);
    close(client_fd);
}
```

**第四步 `main` 核心：**

```cpp
setNonBlocking(listen_fd);
int epfd = epoll_create1(0);
addFd(epfd, listen_fd);

vector<epoll_event> events(MAX_EVENTS);
while (true) {
    int nready = epoll_wait(epfd, events.data(), MAX_EVENTS, -1);
    for (int i = 0; i < nready; i++) {
        int fd = events[i].data.fd;
        if (fd == listen_fd) {
            while (accept ... 直到 EAGAIN)
                addFd(epfd, client_fd);
        } else {
            handleClient(epfd, fd);
        }
    }
}
```

---

### 解答

#### 一、`main` 改了什么？

| 项目 | 第三步 | 第四步 | 为什么 |
|------|--------|--------|--------|
| 等待方式 | `accept` **阻塞**等连接 | `epoll_wait` 等**多种事件** | 同时盯住 listen + 很多 client |
| 监听 socket | 直接 `accept` | 先 `addFd(epfd, listen_fd)` | 新连接也作为一种「可读事件」 |
| 客户端 socket | accept 后立刻 read | accept 后 `addFd`，等 EPOLLIN 再 `handleClient` | 没数据时不占用 accept 后的处理时间 |
| 非阻塞 | 无 | `setNonBlocking` on listen/client | 配合 epoll + accept 循环 |
| 新增变量 | 无 | `epfd`、`events[]`、`MAX_EVENTS` | epoll API 需要 |
| 业务代码位置 | 全在 `main` 里 | 抽到 **`handleClient`** | main 只负责事件分发 |

**第三步的问题（为什么要改）：**

```text
accept 阻塞时：无法处理已连接客户端的后续数据（其实第三步 accept 后立即 read，一次搞定，尚可）
更关键的是：处理 client A 时，accept 不到 B，B 在内核队列里等
用 epoll：A、B、C 谁有数据 epoll_wait 一次返回多个，轮流处理
```

---

#### 二、`handleClient` 是什么？（相对第三步）

第三步**没有** `handleClient`，等价代码在 `main` 的 `read → handleRequest → write → close`。

第四步 **`handleClient` = 把 main 里「单个 client 的处理」原封不动搬出来**，并加上 epoll 清理：

| 步骤 | 第三步（在 main） | 第四步（在 handleClient） |
|------|------------------|-------------------------|
| 读请求 | `read` | `read` ✅ 相同 |
| 解析 | `parseRequestLine` | ✅ 相同 |
| 业务 | `handleRequest` | `buildResponseFromRequest` ✅ 相同逻辑 |
| 写响应 | `write` | ✅ 相同 |
| epoll 删除 | 无 | **`removeFd`** ⭐ 新增 |
| 关连接 | `close` | ✅ 相同 |

---

#### 三、结构对比图

```text
【Step3】
main
 └─ while
      └─ accept（阻塞）
           └─ read / parse / handleRequest / write / close


【Step4】
main
 └─ epoll_create + addFd(listen)
 └─ while
      └─ epoll_wait
           ├─ listen_fd  → while accept → addFd(client)
           └─ client_fd   → handleClient
                                ├─ read / parse
                                ├─ buildResponseFromRequest
                                ├─ write
                                ├─ removeFd
                                └─ close
```

---

#### 四、为什么不让 `main` 里继续写 read/write？

可以写，但会又长又乱：

```cpp
// 若不用 handleClient，main 里 else 分支要写一大坨
} else {
    read(...); parse...; buildResponse...; write...; removeFd...; close...
}
```

抽成 **`handleClient(epfd, fd)`** 后，`main` 的 epoll 循环只有：

```cpp
if (fd == listen_fd) { ... }
else { handleClient(epfd, fd); }
```

**职责分离：**

| 函数 | 职责 |
|------|------|
| `main` | epoll 总调度：等新连接 / 分发客户端 |
| `handleClient` | 一个连接的生命周期 |
| `buildResponseFromRequest` | 纯 HTTP + 静态文件业务 |

---

#### 五、改了什么、没改什么

| 没变 | 变了 |
|------|------|
| `parseRequestLine` | 等待模型：`accept` 阻塞 → **epoll** |
| `buildHttpResponse` | 新增 `setNonBlocking`、`addFd`、`removeFd` |
| `pathToFile` / `readFile` / `getContentType` | 新增 **`handleClient`** |
| 静态文件 `www/` | `main` 循环结构完全重写 |
| 业务响应内容 | listen 也纳入 epoll 监视 |

---

#### 小结

| 你的问题 | 答案 |
|---------|------|
| `main` 为何大改？ | 从「只等 accept」变成「epoll 等所有 fd 事件」 |
| `handleClient` 从哪来？ | 第三步 main 里单连接处理 + `removeFd` |
| 为何这么做？ | 单进程监视多连接，结构清晰，为后续高并发打基础 |

## 问题6：`epoll_create1` 是什么？参数什么意思？

### 相关代码

```cpp
int epfd = epoll_create1(0);
if (epfd < 0) {
    cerr << "epoll_create1 失败\n";
    return 1;
}

addFd(epfd, listen_fd);   // 之后用 epfd 做 ctl / wait
```

### 解答

#### 一句话

**`epoll_create1` 创建一个 epoll「监视器」，返回一个 epoll 专用的文件描述符 `epfd`。**  
后面所有 `epoll_ctl`（加/删 fd）和 `epoll_wait`（等事件）都要用这个 `epfd`。

---

#### 名字怎么读？

| 部分 | 含义 |
|------|------|
| **epoll** | event poll，事件轮询机制 |
| **create** | 创建 |
| **1** | 第二版创建 API（比老的 `epoll_create` 更推荐） |

老接口 `epoll_create(int size)` 里 `size` 参数已被内核忽略；**新项目用 `epoll_create1` 即可**。

---

#### 函数原型

```cpp
int epoll_create1(int flags);
```

| 返回值 | 含义 |
|--------|------|
| `>= 0` | 成功，返回 **epoll 实例的 fd**（常叫 `epfd`） |
| `-1` | 失败，用 `errno` 查原因 |

**注意：** 返回的 `epfd` 本身也是一个 fd，程序结束前应 `close(epfd)`（Step4 正常跑在 `while(true)` 里，很少执行到）。

---

#### 参数 `flags`：你的代码里是 `0`

```cpp
int epfd = epoll_create1(0);
```

`0` 表示：**不使用任何可选标志**，默认行为就够学习用。

常见可选标志（了解即可，本篇未用）：

| 标志 | 含义 |
|------|------|
| `0` | 默认 |
| `EPOLL_CLOEXEC` | 执行 `exec` 新程序时自动关闭 `epfd`，防泄漏 |

示例（生产代码常见写法）：

```cpp
int epfd = epoll_create1(EPOLL_CLOEXEC);
```

---

#### `epfd` 和 `listen_fd` / `client_fd` 是什么关系？

三者**都是 fd（整数编号）**，但角色不同：

| fd | 角色 | 类比 |
|----|------|------|
| **`epfd`** | epoll 监视器本身 | 大厅**值班表**本体 |
| **`listen_fd`** | 监听新连接的 socket | 餐厅**大门** |
| **`client_fd`** | 某一个客户端连接 | 某一**桌客人** |

关系：

```text
epoll_create1()  →  得到 epfd（监视器）
epoll_ctl(epfd, ADD, listen_fd, ...)   // 把大门登记到值班表
epoll_ctl(epfd, ADD, client_fd, ...)   // 把某桌登记到值班表
epoll_wait(epfd, ...)                  // 看值班表：谁有事
```

**`epfd` 不直接用来 read/write HTTP**，只用来管理「监视哪些 socket」。

---

#### 在 Step4 里何时调用？调用几次？

| 时机 | 次数 |
|------|------|
| `listen` 成功之后、`addFd(listen_fd)` 之前 | **整个程序只调用 1 次** |

```text
socket → bind → listen
    → setNonBlocking(listen_fd)
    → epoll_create1(0)     ← 只创建一次
    → addFd(epfd, listen_fd)
    → while { epoll_wait ... }
```

---

#### 生活类比

开餐厅前先准备一本**空的值班登记表**（`epoll_create1`），登记本有一个编号 `epfd`。  
之后每来一扇门（listen）或一桌客（client），都用 `epoll_ctl` 把桌号写进这本表。

---

#### 小结

| 问题 | 答案 |
|------|------|
| `epoll_create1` 干什么？ | 创建 epoll 实例，返回 `epfd` |
| 参数 `0` 是什么？ | 不用额外标志 |
| 返回值是什么？ | epoll 专用的 fd，给 `epoll_ctl` / `epoll_wait` 用 |
| 和 socket fd 一样吗？ | 都是 fd，但 **epfd 只管监视，不直接收发 HTTP** |

---

## 问题7：`epoll_wait` 是什么？参数什么意思？

### 相关代码

```cpp
const int MAX_EVENTS = 64;
vector<epoll_event> events(MAX_EVENTS);

while (true) {
    int nready = epoll_wait(epfd, events.data(), MAX_EVENTS, -1);
    if (nready < 0) {
        cerr << "epoll_wait 失败\n";
        continue;
    }

    for (int i = 0; i < nready; i++) {
        int fd = events[i].data.fd;   // 哪个 fd 就绪了
        if (fd == listen_fd) { ... }
        else { handleClient(epfd, fd); }
    }
}
```

### 解答

#### 一句话

**`epoll_wait` 阻塞等待，直到「监视名单里至少有一个 fd 就绪」，然后把就绪的 fd 列表写进 `events` 数组，返回个数 `nready`。**

Step4 的主循环心脏就是：**`epoll_wait` → 遍历 `nready` 个事件 → 分发处理**。

---

#### 函数原型

```cpp
int epoll_wait(int epfd,
               struct epoll_event *events,
               int maxevents,
               int timeout);
```

---

#### 参数逐个解析

##### 参数 1：`epfd`

| 含义 | 用哪个 epoll 实例等待 |
|------|----------------------|
| 你的代码 | `epoll_create1` 返回的那个 `epfd` |

必须和 `epoll_ctl` 用的是**同一个** epfd。

---

##### 参数 2：`events`（你的代码：`events.data()`）

| 含义 | **输出数组**：内核把就绪事件写进这里 |
|------|--------------------------------------|
| 类型 | `epoll_event` 数组的首指针 |
| 你的写法 | `vector<epoll_event> events(MAX_EVENTS);` 然后传 `events.data()` |

`epoll_wait` 返回后，只有前 **`nready`** 个元素有效：

```cpp
for (int i = 0; i < nready; i++) {
    int fd = events[i].data.fd;      // 我们当初 addFd 时存的
    // events[i].events 里还有 EPOLLIN 等标志（本篇一般不再判断）
}
```

**为什么用 `vector`？** 只是方便分配一块连续内存；也可以用 `epoll_event events[MAX_EVENTS];` 栈数组。

---

##### 参数 3：`maxevents`（你的代码：`MAX_EVENTS`，值为 64）

| 含义 | **`events` 数组最多能装几个事件**（数组容量上限） |
|------|--------------------------------------------------|
| 你的代码 | `MAX_EVENTS = 64` |

**不是**「一次必须等 64 个」，而是：

```text
若同时就绪 3 个 fd  →  nready = 3，events[0..2] 有效
若同时就绪 100 个   →  nready 最多 64（多的下次 wait 再拿）
```

所以要 **`maxevents` ≥ 你预期一次处理的事件数**。学习用 64 足够。

---

##### 参数 4：`timeout`（你的代码：`-1`）

| 值 | 行为 |
|----|------|
| **`-1`** | **一直阻塞**，直到至少有一个 fd 就绪（Step4 用法） |
| `0` | **不阻塞**，立刻返回（轮询模式） |
| `> 0` | 最多等这么多**毫秒**，超时返回 0 |

```cpp
epoll_wait(epfd, events.data(), MAX_EVENTS, -1);
//                                              ↑ 没有事件就一直睡，不空转 CPU
```

---

#### 返回值 `nready`

| 返回值 | 含义 |
|--------|------|
| `> 0` | 有 **`nready` 个** fd 就绪，详情在 `events[0 .. nready-1]` |
| `0` | 超时（仅当 `timeout` 为 0 或正数时会出现） |
| `-1` | 出错（如被信号中断 `EINTR`） |

Step4 里：

```cpp
for (int i = 0; i < nready; i++) { ... }
```

只循环 **`nready` 次**，不是 `MAX_EVENTS` 次。

---

#### 什么叫「就绪」？和 `EPOLLIN` 的关系

`addFd` 时我们设置了 `ev.events = EPOLLIN`（关心可读）。

| fd 类型 | `EPOLLIN` 就绪时通常表示 |
|---------|-------------------------|
| `listen_fd` | 有新连接，可以 `accept` |
| `client_fd` | 有数据可读，可以 `read` |

`epoll_wait` 返回后，说明这些 fd **至少有一个**达到了你订阅的事件（本篇就是可读）。

---

#### 和第三步 `accept` 阻塞的对比

| | 第三步 | 第四步 |
|---|--------|--------|
| 等什么 | 只等**新连接**（`accept`） | 等**新连接 + 已有连接的数据** |
| API | `accept` 阻塞 | `epoll_wait` 阻塞 |
| 返回 | 一个 `client_fd` | 可能**多个**就绪 fd（`nready` 个） |

```text
第三步：accept 睡死 → 只能接新客

第四步：epoll_wait 睡死 → 新客（listen）或老客有数据（client）都会醒
```

---

#### 一次 `epoll_wait` 返回多个 fd 时怎么办？

```text
epoll_wait 返回 nready = 3

events[0].data.fd = listen_fd   → 去 while(accept)
events[1].data.fd = 5           → handleClient(epfd, 5)
events[2].data.fd = 7           → handleClient(epfd, 7)
```

**for 循环逐个处理**即可。处理完再回到 `epoll_wait` 等下一批。

---

#### 完整流程串起来

```text
epoll_create1(0)           → 创建 epfd
epoll_ctl(ADD, listen_fd)  → 监视大门
        │
        ▼
   epoll_wait(-1)  ←──────────┐
        │                     │
        ├─ listen 就绪 → accept → addFd(client)
        ├─ client 就绪 → handleClient → removeFd + close
        │                     │
        └─────────────────────┘
              循环
```

---

#### 生活类比

| 概念 | 类比 |
|------|------|
| `epoll_wait` | 值班员**坐在大厅等**举手 |
| `timeout = -1` | 没人举手就一直等，不瞎跑 |
| `events` 数组 | 举手名单（本轮） |
| `nready` | 本轮有几桌举手 |
| `events[i].data.fd` | 举手的桌号 |

---

#### 小结

| 问题 | 答案 |
|------|------|
| `epoll_wait` 干什么？ | 等监视名单里的 fd 就绪，写入 `events` |
| `epfd` | 监视器实例（`epoll_create1` 的返回值） |
| `events.data()` | 输出数组，装就绪的 `epoll_event` |
| `MAX_EVENTS` | 数组**容量**上限，不是固定等几个 |
| `-1` | 无限等待，直到有事件 |
| `nready` | 本次实际就绪个数，for 循环用它 |

## 附录：第四步函数分工一张表

| 函数 | 层次 | 一句话 |
|------|------|--------|
| `epoll_create1` | epoll 核心 | 创建监视器，返回 `epfd` |
| `epoll_wait` | epoll 核心 | 阻塞等待就绪 fd，写入 `events`，返回 `nready` |
| `setNonBlocking` | 工具 | fd 设为非阻塞 |
| `addFd` | 工具 | fd 加入 epoll，监视 EPOLLIN |
| `removeFd` | 工具 | fd 从 epoll 移除 |
| `buildResponseFromRequest` | 业务 | 等同 Step3 `handleRequest` |
| `handleClient` | I/O + 连接 | read → 业务 → write → DEL → close |
| `main` | 调度 | epoll_wait → accept 或 handleClient |
