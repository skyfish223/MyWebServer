## 第一部分：先搞懂"我们在升级什么"

### 1.1 第四篇的瓶颈在哪？

第四篇用 epoll 同时**监视**很多连接，但 `handleClient` 里仍在一个线程里做完所有事：

```text
epoll_wait 返回 client 可读
    → read
    → parseRequestLine
    → buildResponseFromRequest（读 www 文件、拼 HTTP）  ← 可能较慢
    → write
    → close
```

| 问题 | 说明 |
|------|------|
| **业务阻塞 I/O 线程** | 读大文件、拼响应时，主循环不能及时 `epoll_wait` 处理其它 fd |
| **单核吃满业务** | 静态文件逻辑和 epoll 调度挤在**同一个线程** |
| **多核浪费** | CPU 有 4 核 8 核，进程却主要用 1 个核跑业务 |

对小网站、本机测试仍可用；要更好利用多核，常见做法是 **Reactor（epoll 线程）+ 线程池（业务线程）**。

---

### 1.2 生活类比

| 角色 | 第四篇 | 第五篇 |
|------|--------|--------|
| **前台（epoll 主线程）** | 既迎宾又下厨 | **只迎宾、收单子**（accept、read） |
| **后厨（线程池）** | 没有 | **多个厨师**并行做菜（解析、读文件、拼响应） |
| **传菜** | 前台自己端 | 厨师做好后 **write + close** |

---

### 1.3 本篇架构（Reactor + 线程池）

```text
┌─────────────────────────────────────────────────────────┐
│  主线程（I/O 线程）                                       │
│  epoll_wait → accept / read → 把任务扔进线程池队列        │
│  （不再自己读 www、拼大段响应）                            │
└───────────────────────────┬─────────────────────────────┘
                            │ 任务队列（mutex 保护）
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   工作线程 1           工作线程 2           工作线程 N
   解析 + 读文件        解析 + 读文件        ...
   write + close        write + close
```

**分工原则（本篇）：**

| 谁做 | 做什么 |
|------|--------|
| **主线程** | `epoll_wait`、`accept`、`read` 请求、`removeFd`、**投递任务** |
| **工作线程** | `parseRequestLine`、`buildResponseFromRequest`（读磁盘）、`write`、`close` |

> 为什么 `write` 放工作线程？每个 `client_fd` 彼此独立，不同连接由不同线程 write 是常见、可学习的简化模型。`epoll_wait` 仍只在主线程调用。

---

### 1.4 本篇程序整体流程

```text
  启动
    → 创建 ThreadPool（如 4 个工作线程）
    → socket / bind / listen / epoll（同第四篇）
    → while (true) {
          epoll_wait
          listen_fd 就绪 → accept → addFd(client)
          client_fd 就绪 → read 请求
                         → removeFd（从 epoll 摘掉）
                         → pool.submit( 工作线程处理 )
        }

  工作线程里：
    parseRequestLine → buildResponseFromRequest → write → close
```

---

### 1.5 线程池是什么？

**线程池** = 预先创建好几个工作线程 + 一个**任务队列**：

```text
主线程：pool.submit(任务);   // 往队列里塞活
工作线程：循环从队列取任务 → 执行
```

好处：

- 不用每来一个请求就 `new thread`（创建线程很贵）
- 线程数可控（如 4 个），不会无限增长
- 多核可以**并行**处理多个请求的「读文件 + 拼响应」

## 第二部分：需要哪些新"工具"，为什么要用

### 2.1 新增头文件

| 头文件 | 提供什么 | 为什么需要 |
|--------|---------|-----------|
| `<thread>` | `std::thread` | 创建工作线程 |
| `<mutex>` | `std::mutex`、`std::lock_guard`、`std::unique_lock` | 保护任务队列，防止多线程同时改队列 |
| `<condition_variable>` | `std::condition_variable` | 队列空时工作线程睡眠，有任务时唤醒 |
| `<queue>` | `std::queue` | 存放待执行的任务 |
| `<functional>` | `std::function` | 用函数/ lambda 表示「一个任务」 |

第四篇已有头文件**全部保留**。

### 2.2 编译注意

链接 pthread（Linux 下通常需要）：

```bash
g++ -std=c++17 -Wall -pthread -o server Step1to6/Step5.cpp
```

---

### 2.3 新增：`ThreadPool` 类

| 成员 | 作用 |
|------|------|
| `queue<function<void()>> tasks` | 任务队列 |
| `mutex mtx` | 队列互斥锁 |
| `condition_variable cv` | 等待/通知 |
| `vector<thread> workers` | 工作线程 |
| `bool stop` | 析构时通知线程退出 |
| `submit(task)` | 主线程往队列里塞任务 |

### 2.4 新增 / 改造函数

| 函数 | 职责 |
|------|------|
| **`processRawRequest(raw)`** | 解析 + `buildResponseFromRequest`，返回响应字符串（给工作线程用） |
| **`dispatchClient(epfd, client_fd, pool)`** | 主线程：read → removeFd → submit 到线程池 |
| ~~`handleClient`~~ | 拆成 **dispatchClient（主线程）** + **lambda 任务（工作线程）** |

### 2.5 同步原语（够用版）

| 概念 | 一句话 |
|------|--------|
| **mutex（互斥锁）** | 同一时刻只允许一个线程进临界区（如操作 `tasks` 队列） |
| **condition_variable** | 「队列空了睡吧」「有新任务了起床」 |
| **lock_guard / unique_lock** | RAII 方式加锁，出了作用域自动解锁 |

## 第三部分：完整代码 + 逐行讲解

### 3.1 完整代码

保存为 **`Step1to6/Step5.cpp`**：

```cpp
#include <iostream>
#include <fstream>
#include <sstream>
#include <string>
#include <vector>
#include <queue>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <functional>
#include <unistd.h>
#include <fcntl.h>
#include <cerrno>
#include <sys/epoll.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <arpa/inet.h>

using namespace std;

const string WEB_ROOT = "www";
const int MAX_EVENTS = 64;
const int PORT = 8080;
const size_t THREAD_POOL_SIZE = 4;

// ========== 第二～四篇：HTTP + 静态文件（保持不变）==========
struct HttpRequest {
    string method;
    string path;
    string version;
};

bool parseRequestLine(const string& raw, HttpRequest& req) {
    size_t lineEnd = raw.find("\r\n");
    if (lineEnd == string::npos) lineEnd = raw.find('\n');
    if (lineEnd == string::npos) return false;
    string line = raw.substr(0, lineEnd);
    istringstream iss(line);
    if (!(iss >> req.method >> req.path >> req.version)) return false;
    return true;
}

string buildHttpResponse(int statusCode, const string& statusText,
                         const string& contentType, const string& body) {
    return "HTTP/1.1 " + to_string(statusCode) + " " + statusText + "\r\n"
           "Content-Type: " + contentType + "\r\n"
           "Content-Length: " + to_string(body.size()) + "\r\n"
           "Connection: close\r\n"
           "\r\n" + body;
}

string pathToFile(const string& urlPath) {
    if (urlPath == "/" || urlPath == "/index.html")
        return WEB_ROOT + "/index.html";
    return WEB_ROOT + urlPath;
}

bool isPathSafe(const string& urlPath) {
    return urlPath.find("..") == string::npos;
}

string getContentType(const string& filePath) {
    if (filePath.size() >= 5 && filePath.substr(filePath.size() - 5) == ".html")
        return "text/html; charset=utf-8";
    if (filePath.size() >= 4 && filePath.substr(filePath.size() - 4) == ".css")
        return "text/css; charset=utf-8";
    if (filePath.size() >= 3 && filePath.substr(filePath.size() - 3) == ".js")
        return "application/javascript; charset=utf-8";
    if (filePath.size() >= 4 && filePath.substr(filePath.size() - 4) == ".png")
        return "image/png";
    if (filePath.size() >= 4 && filePath.substr(filePath.size() - 4) == ".jpg")
        return "image/jpeg";
    return "application/octet-stream";
}

bool readFile(const string& filePath, string& out) {
    ifstream ifs(filePath, ios::binary);
    if (!ifs) return false;
    out.assign((istreambuf_iterator<char>(ifs)),
               istreambuf_iterator<char>());
    return true;
}

string buildResponseFromRequest(const HttpRequest& req) {
    if (req.method != "GET")
        return buildHttpResponse(405, "Method Not Allowed",
            "text/plain; charset=utf-8", "暂只支持 GET");
    if (!isPathSafe(req.path))
        return buildHttpResponse(403, "Forbidden",
            "text/plain; charset=utf-8", "403 - 非法路径");
    string filePath = pathToFile(req.path);
    string body;
    if (!readFile(filePath, body))
        return buildHttpResponse(404, "Not Found",
            "text/plain; charset=utf-8", "404 - 文件不存在: " + req.path);
    return buildHttpResponse(200, "OK", getContentType(filePath), body);
}

// 工作线程：从原始报文生成响应（解析 + 读文件 + 拼 HTTP）
string processRawRequest(const string& rawRequest) {
    HttpRequest req;
    if (!parseRequestLine(rawRequest, req))
        return buildHttpResponse(400, "Bad Request",
            "text/plain; charset=utf-8", "请求行格式错误");
    cout << "[worker " << this_thread::get_id() << "] "
         << req.method << " " << req.path << "\n";
    return buildResponseFromRequest(req);
}

// ========== 第五篇：线程池 ==========
class ThreadPool {
public:
    explicit ThreadPool(size_t threadCount) : stop_(false) {
        for (size_t i = 0; i < threadCount; ++i) {
            workers_.emplace_back([this]() { workerLoop(); });
        }
    }

    ~ThreadPool() {
        {
            lock_guard<mutex> lock(mtx_);
            stop_ = true;
        }
        cv_.notify_all();
        for (auto& t : workers_) {
            if (t.joinable()) t.join();
        }
    }

    void submit(function<void()> task) {
        {
            lock_guard<mutex> lock(mtx_);
            tasks_.push(move(task));
        }
        cv_.notify_one();
    }

private:
    void workerLoop() {
        while (true) {
            function<void()> task;
            {
                unique_lock<mutex> lock(mtx_);
                cv_.wait(lock, [this]() { return stop_ || !tasks_.empty(); });
                if (stop_ && tasks_.empty()) return;
                task = move(tasks_.front());
                tasks_.pop();
            }
            task();
        }
    }

    queue<function<void()>> tasks_;
    mutex mtx_;
    condition_variable cv_;
    vector<thread> workers_;
    bool stop_;
};

// ========== 第四篇：epoll 工具（保持不变）==========
void setNonBlocking(int fd) {
    int flags = fcntl(fd, F_GETFL, 0);
    fcntl(fd, F_SETFL, flags | O_NONBLOCK);
}

void addFd(int epfd, int fd) {
    epoll_event ev{};
    ev.events = EPOLLIN;
    ev.data.fd = fd;
    epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);
}

void removeFd(int epfd, int fd) {
    epoll_ctl(epfd, EPOLL_CTL_DEL, fd, nullptr);
}

// ========== 第五篇：主线程只 read + 投递任务 ==========
void dispatchClient(int epfd, int client_fd, ThreadPool& pool) {
    char buffer[8192] = {0};
    ssize_t n = read(client_fd, buffer, sizeof(buffer) - 1);

    removeFd(epfd, client_fd);

    string rawRequest;
    if (n > 0) rawRequest.assign(buffer, n);

    pool.submit([client_fd, rawRequest, n]() {
        string response;
        if (n <= 0) {
            response = buildHttpResponse(400, "Bad Request",
                "text/plain; charset=utf-8", "无法读取请求");
        } else {
            response = processRawRequest(rawRequest);
        }
        write(client_fd, response.c_str(), response.size());
        close(client_fd);
        cout << "[worker] 完成 fd=" << client_fd << "\n";
    });
}

int main() {
    ThreadPool pool(THREAD_POOL_SIZE);

    int listen_fd = socket(AF_INET, SOCK_STREAM, 0);
    if (listen_fd < 0) {
        cerr << "socket 失败\n";
        return 1;
    }

    int opt = 1;
    setsockopt(listen_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    sockaddr_in addr{};
    addr.sin_family = AF_INET;
    addr.sin_addr.s_addr = INADDR_ANY;
    addr.sin_port = htons(PORT);

    if (bind(listen_fd, (sockaddr*)&addr, sizeof(addr)) < 0) {
        cerr << "bind 失败\n";
        return 1;
    }

    if (listen(listen_fd, 128) < 0) {
        cerr << "listen 失败\n";
        return 1;
    }

    setNonBlocking(listen_fd);

    int epfd = epoll_create1(0);
    if (epfd < 0) {
        cerr << "epoll_create1 失败\n";
        return 1;
    }

    addFd(epfd, listen_fd);

    cout << "线程池服务器已启动：http://127.0.0.1:" << PORT << "\n";
    cout << "工作线程数: " << THREAD_POOL_SIZE << "\n";
    cout << "网站根目录: " << WEB_ROOT << "/\n";

    vector<epoll_event> events(MAX_EVENTS);

    while (true) {
        int nready = epoll_wait(epfd, events.data(), MAX_EVENTS, -1);
        if (nready < 0) {
            cerr << "epoll_wait 失败\n";
            continue;
        }

        for (int i = 0; i < nready; ++i) {
            int fd = events[i].data.fd;

            if (fd == listen_fd) {
                while (true) {
                    sockaddr_in client_addr{};
                    socklen_t len = sizeof(client_addr);
                    int client_fd = accept(listen_fd,
                                           (sockaddr*)&client_addr, &len);
                    if (client_fd < 0) {
                        if (errno == EAGAIN || errno == EWOULDBLOCK) break;
                        cerr << "accept 失败\n";
                        break;
                    }
                    setNonBlocking(client_fd);
                    addFd(epfd, client_fd);
                    cout << "[main] 新连接 fd=" << client_fd << "\n";
                }
            } else {
                dispatchClient(epfd, fd, pool);
            }
        }
    }

    close(epfd);
    close(listen_fd);
    return 0;
}
```

### 3.2 逐块讲解：线程池

#### `ThreadPool` 构造函数：创建工作线程

```cpp
for (size_t i = 0; i < threadCount; ++i) {
    workers_.emplace_back([this]() { workerLoop(); });
}
```

启动 `THREAD_POOL_SIZE`（如 4）个线程，每个线程执行 `workerLoop()`：**无限循环取任务执行**。

---

#### `submit`：主线程投递任务

```cpp
void submit(function<void()> task) {
    {
        lock_guard<mutex> lock(mtx_);
        tasks_.push(move(task));
    }
    cv_.notify_one();
}
```

| 步骤 | 含义 |
|------|------|
| `lock_guard` | 加锁，保护 `tasks_` 队列 |
| `tasks_.push` | 把任务放进队列 |
| `cv_.notify_one()` | 叫醒**一个**正在睡觉的工作线程 |

---

#### `workerLoop`：工作线程取任务

```cpp
cv_.wait(lock, [this]() { return stop_ || !tasks_.empty(); });
```

| 部分 | 含义 |
|------|------|
| `cv_.wait` | 队列空则睡眠，避免空转占 CPU |
| `stop_ \|\| !tasks_.empty()` | 有任务或要退出时才醒 |
| `task()` | 执行任务（lambda 里的业务逻辑） |

---

### 3.3 逐块讲解：`dispatchClient`

第四篇 `handleClient` 在主线程里做完所有事；第五篇拆开：

```cpp
void dispatchClient(int epfd, int client_fd, ThreadPool& pool) {
    read(...);                    // 主线程读（快）
    removeFd(epfd, client_fd);    // 主线程从 epoll 摘掉
    pool.submit([...]() {         // 慢活交给工作线程
        response = processRawRequest(rawRequest);
        write(...);
        close(client_fd);
    });
}
```

**为什么先 `removeFd`？**  
请求已 `read` 进内存，这个 fd 在 epoll 里再触发 `EPOLLIN` 没意义；摘掉后由工作线程 `close`。

**lambda 捕获了什么？**

```cpp
pool.submit([client_fd, rawRequest, n]() { ... });
```

| 捕获 | 原因 |
|------|------|
| `client_fd` | 工作线程要 write/close |
| `rawRequest` | 请求内容拷贝进任务（主线程继续跑 epoll 不能共用栈上 buffer） |
| `n` | 判断 read 是否成功 |

---

### 3.4 `processRawRequest`：工作线程里的业务

```cpp
string processRawRequest(const string& rawRequest) {
    parseRequestLine → buildResponseFromRequest
}
```

等价于第四篇 `handleClient` 里**解析 + 读 www** 那一段，只是放到工作线程执行。

`this_thread::get_id()` 打印线程 id，方便你在终端看到**不同 worker** 在处理请求。

---

### 3.5 与第四篇对照表

| 项目 | 第四篇 | 第五篇 |
|------|--------|--------|
| 事件循环 | epoll 主线程 | 同左 |
| read 请求 | `handleClient` 内 | **`dispatchClient`（主线程）** |
| 解析 + 读文件 | `handleClient` 内 | **`processRawRequest`（工作线程）** |
| write + close | `handleClient` 内 | **工作线程** |
| 并发执行业务 | 串行 | **最多 THREAD_POOL_SIZE 个并行** |
| 新组件 | 无 | **`ThreadPool`** |

## 常见问题排查

### Q1：编译报错 `undefined reference to pthread_create`

加 **`-pthread`**：

```bash
g++ -std=c++17 -Wall -pthread -o server Step1to6/Step5.cpp
```

### Q2：页面空白或连接重置

- 检查 lambda 是否捕获了 `rawRequest`（按值），而不是悬空指针  
- 检查 `write` 是否在 `close` 之前  
- 检查 `!isPathSafe` 是否写反（见 Step4 的 403 问题）

### Q3：感觉和 Step4 一样快

本机少量请求差别不大；线程池优势在**多请求同时做磁盘 I/O 和拼响应**时更明显。可用上面 for 循环多 `curl` 观察 worker 日志。

### Q4：任务队列会无限增长吗？

会。本篇未做「队列上限」和背压，学习用足够。生产环境需限制队列长度或拒绝新连接。

### Q5：`epoll_wait` 和工作线程会冲突吗？

本篇约定：**只有主线程**调用 `epoll_wait` / `epoll_ctl`；工作线程只 `write`/`close` 自己的 `client_fd`，不碰 `epfd`。

### Q6：想改线程数

改 `THREAD_POOL_SIZE`，一般设为 CPU 核心数或 2×核心数即可，不必太多。
